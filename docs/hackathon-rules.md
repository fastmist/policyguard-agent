# Hedera Apex Hackathon - 评审规则与标准

> **适用赛道**: AI & Agents / DeFi & Tokenization / Sustainability / Open Track  
> **不适用**: Legacy Builders (有单独标准)

---

## 📊 评审维度权重

| 维度 | 权重 | 核心问题 |
|------|------|----------|
| **Innovation 创新** | 10% | 是否新颖？Hedera生态中是否已有？ |
| **Feasibility 可行性** | 10% | 能否用Hedera实现？团队是否理解问题域？ |
| **Execution 执行** | 20% | 是否完成MVP？团队协作如何？UX如何？ |
| **Integration 集成** | 15% | Hedera网络使用程度？服务集成是否创新？ |
| **Success 成功潜力** | 20% | 对Hedera网络/生态的正面影响？ |
| **Validation 验证** | 15% | 市场反馈循环？早期用户/收入？ |
| **Pitch 路演** | 10% | 问题&方案是否清晰？故事性如何？ |

---

## 🔍 各维度详细标准

### 1. Innovation 创新 (10%)

**评审要点**:
- [ ] 项目是否符合所选赛道主题？
- [ ] 解决方案的创新程度？跨链是否已有类似？
- [ ] Hedera生态中是否已有类似项目？
- [ ] 是否扩展或建立了Hedera生态的新能力？

**加分项**:
- 全新概念，Hedera生态首创
- 跨链迁移的创新适配
- 解决之前未被解决的问题

---

### 2. Feasibility 可行性 (10%)

**评审要点**:
- [ ] 能否使用Hedera网络服务或生态平台实现？
- [ ] 是否必须是Web3方案？还是Web2也能做？
- [ ] 团队是否理解所解决问题的问题域？
- [ ] 团队是否理解执行方案所需条件，并具备执行能力？
- [ ] 是否创建并理解了精益/商业模式画布？

**关键文档**:
- 商业模式画布 (Lean/Business Model Canvas)
- 技术可行性分析
- 风险评估

---

### 3. Execution 执行 (20%)

**评审要点**:
- [ ] 团队是否创建了MVP？
- [ ] 是否交付了具有关键功能的PoC？
- [ ] 是否创建了完全可用的解决方案？
- [ ] 团队协作是否良好？是否发挥各自优势？团队领导是否有效？
- [ ] 是否为解决方案制定了超越黑客松的长期开发策略和计划？
- [ ] 是否识别了GTM (Go-To-Market) 策略？
- [ ] 是否识别了通过市场反馈循环改进方案/想法的机会？
- [ ] 是否识别了重要的设计决策？
- [ ] 是否重视用户体验/可访问性？

**交付物要求**:
- 可运行的代码
- 清晰的README
- 部署/测试说明
- 技术架构文档

---

### 4. Integration 集成 (15%)

**评审要点**:
- [ ] 项目在多大程度上使用了Hedera网络？
- [ ] 集成深度如何？使用了哪些服务？
- [ ] 是否使用或利用了生态平台？
- [ ] Hedera服务是否以创新的方式集成？（创造性）

**Hedera服务集成**:
- **Hedera Consensus Service (HCS)**: 消息队列、事件日志
- **Hedera Token Service (HTS)**: 代币发行、管理
- **Hedera Smart Contract Service**: EVM兼容智能合约
- **Hedera File Service**: 文件存储
- **Mirror Node**: 数据查询

**加分项**:
- 多服务组合使用
- 创新性的服务应用方式
- 生态平台深度集成

---

### 5. Success 成功潜力 (20%)

**评审要点**:
- [ ] 解决方案是否对Hedera网络或生态产生正面影响？
- [ ] 是否会导致更多Hedera账户创建？
- [ ] 是否会产生大量月度活跃Hedera账户？
- [ ] 是否会增加Hedera网络的TPS？
- [ ] 是否会让更多受众了解Hedera网络？

**量化指标**:
- 潜在用户规模
- 预计交易量
- 网络效应潜力
- 生态扩展性

---

### 6. Validation 验证 (15%)

**评审要点**:
- [ ] 团队是否识别了获取市场反馈的渠道？（如询问用户/客户、合作伙伴、从业者、领域专家等）
- [ ] 是否建立了市场反馈循环？（如跟进试用产品的用户获取反馈、评价）
- [ ] 团队在市场上取得了什么进展？
- [ ] 早期采用者 | 免费/付费试用？ | 是否产生收入？ | 市场 sentiment如何？ | 增长/流失率？

**验证方式**:
- 用户访谈记录
- 调研数据
- 早期用户注册
- 等待列表
- 意向书 (LOI)

---

### 7. Pitch 路演 (10%)

**评审要点**:
- [ ] 问题和解决方案是否清晰呈现？
- [ ] 问题是否足够大，能支撑持续增长？
- [ ] 路演风格和叙事是否经过深思熟虑和执行？
- [ ] 团队是否能传达重大且令人兴奋的机会？
- [ ] 解决方案的数据/指标是否合理？
- [ ] Hedera在路演中如何呈现？
- [ ] 对问题的回答是否恰当？
- [ ] 是否清楚说明MVP功能及选择原因？

**Pitch Deck 建议结构**:
1. **团队与项目介绍**
2. **项目摘要**: 全面概述解决方案（围绕7个评审标准）
3. **未来路线图**: 关键学习点和改进空间？下一步计划？
4. **Demo**: 展示技术实力、可用性和性能（**预录视频上传到YouTube**）

## 🎯 AI & Agents 赛道特殊考量

### 赛道主题

探索AI驱动代理与去中心化基础设施的融合，创建市场、协调层和工具——自主行动者可以思考、交易和协作——利用Hedera的快速、低成本微交易和安全共识，解锁透明、自主经济的崛起。

### 关键问题

- Agent如何在没有人工干预的情况下进行交易？
- 如何建立Agent之间的信任机制？
- 如何利用Hedera的低费用支持高频Agent交互？
- Agent如何发现和协调？
- 如何创建Agent经济系统？

### 参考方向

1. **Agent市场**: Agent雇佣其他Agent、Agent资本池
2. **Agent协调**: Agent竞争任务、Agent订阅其他Agent的输出
3. **Agent组织**: Agent DAOs / 公会 / 集体 / 社区
4. **Agent发现**: 注册表、声誉系统、信任机制

## ⚖️ Legacy Builders 评审标准

（仅适用于曾参与过往届Hedera黑客松的回归参与者）

### 1. Baseline Quality 基线质量 (30%)

- 起始点是否真实且一致？
- 基线Hedera契合度 (10%)
- 基线可行性 (10%)
- 基线产品清晰度 (10%)

### 2. Delta Score 增量分数 (70%)

- 自基线以来项目取得了多少进展？
- 产品范围和UX增量 (25%)
- 技术健壮性增量 (15%)
- 市场验证/验证增量 (15%)
- Hedera集成深度增量 (15%)
